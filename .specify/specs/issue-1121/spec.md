## Design reference
- **Design doc**: `docs/design/15-production-readiness.md`
- **Section**: `¬ß Future ‚Äî No per-step execution timeout`
- **Implements**: Per-step execution timeout via `stepTimeoutSeconds` (üî≤ ‚Üí ‚úÖ)

---

## Zone 1 ‚Äî Obligations (falsifiable, must satisfy)

1. **O1 ‚Äî API field exists**: `Pipeline.spec.environments[].stepTimeoutSeconds` is a new
   optional integer field (default 300 seconds). The field has kubebuilder validation
   `Minimum=1`. When not set, the behavior is identical to before this change (no timeout).

2. **O2 ‚Äî StepState propagation**: `StepState.StepTimeoutSeconds` is a new integer field.
   The reconciler copies `env.StepTimeoutSeconds` to `state.StepTimeoutSeconds` when
   building the state for step execution.

3. **O3 ‚Äî Context timeout in ExecuteFrom**: The `Engine.ExecuteFrom` function creates a
   per-step `context.WithTimeout` when `state.StepTimeoutSeconds > 0`. The derived context
   is passed to `step.Execute`. If the timeout fires, `ctx.Err()` propagates as a fatal
   error causing the PromotionStep to transition to Failed.

4. **O4 ‚Äî Default behavior unchanged**: When `stepTimeoutSeconds` is not set (0), no
   timeout is applied. All existing behavior is unchanged.

5. **O5 ‚Äî Test coverage**: A table-driven test in `pkg/steps/engine_test.go` (or new file)
   verifies that a step that blocks is cancelled when the timeout fires. Uses a mock step
   that blocks on a channel.

---

## Zone 2 ‚Äî Implementer's judgment

- Default value of 300s is appropriate for most operations. Users can set to 0 to disable.
- The timeout applies per-step, not per-step-sequence. A 5-step pipeline with 300s timeout
  has a total maximum execution time of 5√ó300 = 1500s, not 300s.
- `context.WithTimeout` is the correct mechanism (not a goroutine/channel pattern).
- The cleanup cancel function must be called via `defer` to prevent context leak.

---

## Zone 3 ‚Äî Scoped out

- Does NOT modify the Pipeline webhook validation (the zero-value means "no timeout")
- Does NOT add per-step-name different timeouts (uniform per environment)
- Does NOT surface the timeout in PromotionStep status messages (out of scope)
- The CRD YAML is regenerated by kubebuilder but the PR does not need to re-run
  kubebuilder ‚Äî the existing `zz_generated.deepcopy.go` will be manually updated
