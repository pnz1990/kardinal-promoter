## Stage 1: CRD Types and Validation

**Goal:** Define the complete Go types and OpenAPI validation schemas for all user-facing CRDs. This stage produces no running controller — only the schema artifacts that every downstream stage depends on.

### Deliverables

- `Pipeline` CRD type with:
  - `spec.environments[]` (name, gitRepo, branch, path, approvalMode `auto|pr-review`, updateStrategy `kustomize|helm`, healthAdapter, deliveryDelegate, dependsOn)
  - `spec.policyGates[]` (reference by name/namespace)
  - `status.conditions[]`, `status.phase`
- `Bundle` CRD type with:
  - `spec.type` (`image|config|mixed`)
  - `spec.images[]` (repository, tag, digest)
  - `spec.configRef` (gitRepo, commitSHA)
  - `spec.provenance` (commitSHA, ciRunURL, author, timestamp)
  - `spec.intent` (targetEnvironment, skipEnvironments[])
  - `status.phase` (`Available|Promoting|Verified|Failed|Superseded`)
  - `status.environments[]` per-environment evidence (gateResults, approver, mergedAt, healthCheckedAt)
- `PolicyGate` CRD type with:
  - `spec.expression` (CEL expression string)
  - `spec.message` (human-readable description shown when gate blocks)
  - `spec.recheckInterval`
  - `spec.skipPermission` (bool)
  - `spec.selector` (environment label selector for org-level auto-injection)
  - `status.lastEvaluatedAt`, `status.ready` (bool), `status.reason`
- `PromotionStep` CRD type (controller-internal, not user-created) with per-step state and output accumulator
- All types have `+kubebuilder:validation` markers for required fields, enums, and formats
- Generated CRD YAML passes `kubectl apply --dry-run=server` against a kind cluster with no errors
- Kubebuilder sample manifests in `config/samples/` for each CRD

## Stage 3: Graph Generation and kro Integration

**Goal:** Implement the Pipeline-to-Graph translator. For a given Bundle+Pipeline pair, the controller generates a kro `Graph` spec with correctly ordered `PromotionStep` and `PolicyGate` nodes, with dependency edges derived from CEL references.

### Deliverables

- `pkg/graph` package with:
  - `Builder` struct: takes a `Pipeline` and a list of applicable `PolicyGate` objects, returns a `kro.run/v1alpha1/Graph` spec
  - Node generation: one `PromotionStep` node per environment, one `PolicyGate` node per gate per environment
  - Sequential default edges (env[0] → env[1] → ...) and parallel edges from `dependsOn`
  - Org-level gate injection: gates in `platform-policies` namespace with matching selector are prepended to each environment's gate list; teams cannot remove them
  - `ownerReferences` on the Graph pointing to the Bundle (cascade delete on Bundle GC)
- `BundleReconciler` extended to:
  - On `status.phase = Available`, call `graph.Builder` and create the `Graph` object if not already present
  - Set `status.phase = Promoting` and record graph name in status
  - Watch for Graph status changes and reflect them in Bundle status
- Unit tests for `graph.Builder` covering: single-env linear pipeline, 3-env linear pipeline, parallel fan-out with `dependsOn`, org-level gate injection, missing gate reference (error)
- `examples/quickstart/graph-expected.yaml` showing the expected generated Graph for the quickstart Pipeline

### Dependencies

- Stage 2
- kro Graph controller CRD installed in cluster (tested against kro experimental branch)

### Acceptance Criteria

- For the quickstart example (3-env linear pipeline with 1 PolicyGate), the generated Graph has exactly 3 `PromotionStep` nodes and 1 `PolicyGate` node with correct dependency edges
- `kubectl get graph -n <bundle-namespace>` shows the generated Graph within 5 seconds of Bundle creation
- Deleting the Bundle cascades to Graph deletion within 30 seconds
- Unit tests for `graph.Builder` achieve 90% line coverage
- `make test` passes

## Stage 4: PolicyGate CEL Evaluator

**Goal:** Implement the `PolicyGateReconciler` that evaluates CEL expressions against a populated context. Gates block or allow promotion advancement at their position in the Graph.

### Deliverables

- `pkg/cel` package with:
  - `Evaluator` struct wrapping `google/cel-go`
  - `BundleContext` struct providing all CEL variables: `bundle.type`, `bundle.images`, `bundle.provenance`, `bundle.intent`, `schedule.isWeekend`, `schedule.hour`, `schedule.dayOfWeek`, `environment.name`
  - `Evaluate(expr string, ctx BundleContext) (bool, string, error)` — returns pass/fail, reason string, and error
  - Benchmark: `Evaluate` completes in under 10ms for a 10-clause expression
- `PolicyGateReconciler` that:
  - Watches `PolicyGate` objects referenced in active Graphs
  - On each reconcile, builds `BundleContext` from the owning Bundle's status
  - Calls `Evaluator.Evaluate` and patches `status.ready` (bool) and `status.lastEvaluatedAt`
  - Requeues after `spec.recheckInterval` for time-based gates
  - Marks the associated Graph node as passed or blocked
- Built-in gate templates in `config/samples/gates/`:
  - `no-weekend-deploys.yaml`: blocks when `schedule.isWeekend == true`
  - `business-hours-only.yaml`: blocks when `schedule.hour < 9 || schedule.hour > 17`
  - `upstream-soak-required.yaml`: blocks until `bundle.status.environments[upstream].healthCheckedAt` is older than 30 minutes (Phase 2 prerequisite structure)

### Dependencies

- Stage 3

### Acceptance Criteria

- Apply `no-weekend-deploys.yaml` gate and a Bundle on a Saturday: `status.ready = false`, `status.reason` contains "weekend"
- Apply same gate on a Tuesday: `status.ready = true` within 5 seconds
- Gate with `recheckInterval: 1m` re-evaluates without manual trigger
- `cel.Evaluate` benchmark passes: p99 under 10ms on 1000 iterations
- Org-level gate in `platform-policies` namespace cannot be removed by patching a Pipeline in another namespace (RBAC test)
- `make test` passes

- Built-in promotion steps in `pkg/steps/`:
  - `git-clone`: clones the target repo at the environment path
  - `kustomize-set-image`: runs `kustomize edit set image` for each image in the Bundle

  - Validates `X-Hub-Signature-256`

## Stage 6: PromotionStep Reconciler and Full Promotion Loop

**Goal:** Wire the full promotion loop end-to-end. A Bundle created against a 3-environment Pipeline with a PolicyGate flows from `Available` through `Promoting` to `Verified`, opening PRs, running health checks (stubbed), and evaluating gates at each step.

### Deliverables

- `PromotionStepReconciler` that:
  - Watches `PromotionStep` objects created by kro Graph
  - Reads the step type from the Graph node spec and dispatches to the corresponding `steps.Step` implementation
  - Passes outputs between steps via an accumulator stored in `PromotionStep.status.outputs`
  - Sets `PromotionStep.status.phase` to `Running | Succeeded | Failed | Blocked`
  - On `Failed`, sets the owning Bundle's phase to `Failed` with reason
- Default step sequence inferred from Pipeline environment config:
  - `approvalMode: pr-review` → `[git-clone, kustomize-set-image, kustomize-build, git-commit, git-push, open-pr, wait-for-merge, health-check]`
  - `approvalMode: auto` → `[git-clone, kustomize-set-image, kustomize-build, git-commit, git-push, health-check]`
- `health-check` step stubbed to succeed immediately in Stage 6 (real adapters in Stage 7)
- Bundle phase machine: `Available → Promoting → Verified | Failed | Superseded`
- Bundle supersession: when a new Bundle targeting the same Pipeline is created, older Bundles in `Promoting` are set to `Superseded` and their Graphs are deleted
- `kardinal explain` CLI command reading live PromotionStep and PolicyGate statuses:
  ```
  Environment  Step           Status    Reason
  test         kustomize-set-image  Succeeded
  test         open-pr        Succeeded  PR #42
  test         wait-for-merge Succeeded  merged by alice
  uat          open-pr        Running    PR #43 open
  prod         PolicyGate     Blocked    no-weekend-deploys: weekend
  ```
- Quickstart example updated: `examples/quickstart/` end-to-end walkthrough produces a working promotion

### Dependencies

- Stage 5

### Acceptance Criteria

- E2E test: kind cluster + GitHub repo + 3-environment Pipeline + `no-weekend-deploys` gate
  1. Create Bundle on a weekday
  2. Bundle reaches `Verified` with all environments promoted within 5 minutes (using auto-merge in test)
  3. Create Bundle on a weekend (or with mocked schedule)
  4. Bundle stops at prod gate with `status.phase = Promoting`, prod PromotionStep `Blocked`
  5. `kardinal explain` output matches expected table
- Bundle supersession: creating a second Bundle while first is promoting sets first to `Superseded`
- All reconcilers are idempotent: deleting and re-creating a PromotionStep does not duplicate PRs
- `make test` passes

## Stage 7: Health Adapters

**Goal:** Replace the health-check stub with real health verification. The controller checks Deployment readiness, Argo CD Application health+sync, or Flux Kustomization Ready — whichever is present.

### Deliverables

- `pkg/health` package with:
  - `Adapter` interface: `Check(ctx, env Environment) (HealthResult, error)` where `HealthResult` carries phase, message, and checked-at timestamp
  - `DeploymentAdapter`: watches `apps/v1/Deployment` readiness conditions in the target namespace
  - `ArgoCDAdapter`: watches `argoproj.io/v1alpha1/Application` health (`Healthy`) and sync (`Synced`) status
  - `FluxAdapter`: watches `kustomize.toolkit.fluxcd.io/v1/Kustomization` `Ready` condition
  - `AutoDetector`: on controller start and every 5 minutes, lists installed CRDs; selects adapter by priority: `argocd > flux > resource`
  - Remote cluster support: adapter can be configured with a kubeconfig Secret reference; uses `rest.Config` from the Secret data
- `health-check` step: calls the auto-detected adapter; polls every 10 seconds until `Healthy` or until timeout (default 30 minutes, configurable via Pipeline `spec.environments[].healthTimeout`)
- Health result written to `Bundle.status.environments[].healthCheckedAt` for upstream soak time calculation in PolicyGates
- `examples/quickstart/` updated with Argo CD Application resource and health adapter config

### Dependencies

- Stage 6

### Acceptance Criteria

## Stage 13: Rollback and Pause/Resume

**Goal:** Implement `kardinal rollback` as a forward promotion of a prior Bundle and `kardinal pause/resume` with graceful gate behavior. Includes automatic rollback on consecutive health-check failures.

### Deliverables

- Rollback flow:
  - `kardinal rollback <pipeline> --env <environment> [--to version]` creates a new Bundle with `spec.provenance.rollbackOf` pointing to the original Bundle
  - Rollback Bundle has label `kardinal.io/rollback: "true"`; `kardinal/rollback` label applied to its PRs
  - Emergency rollback: `--emergency` flag sets `kardinal.io/emergency: "true"` label; PolicyGate evaluator skips gates where `spec.skipPermission: true`
  - RBAC: emergency rollback requires `create` verb on a `rollbackemergency` resource (custom verb enforced via webhook)
- Automatic rollback:
  - `Pipeline.spec.environments[].autoRollback.failureThreshold` (default: 3 consecutive health-check failures)
  - Controller creates a rollback Bundle automatically; records reason in Bundle status
- Pause/resume:
  - `kardinal pause <pipeline>` injects a PolicyGate with `expression: "false"` into the Pipeline's effective gate list; blocks all new promotion steps immediately
  - Existing PRs remain open; no new steps are started
  - `kardinal resume <pipeline>` deletes the freeze PolicyGate; reconciler re-queues all in-flight steps
- `kardinal rollback` and `kardinal pause` documented in `docs/rollback.md`

## Stage 14: Distributed Mode (Control Plane + Agents)

**Goal:** Split the single binary into a control plane (Pipeline reconciliation, Graph generation, PolicyGate evaluation) and an agent (PromotionStep reconciliation per shard). Agents connect outbound to the control plane.

### Deliverables

- `cmd/kardinal-agent/` binary:
  - Connects to control plane via gRPC over mTLS (control plane address via `--control-plane-address` flag)
  - Watches `PromotionStep` objects labeled with `kardinal.io/shard: <shard-id>` only
  - Reconciles steps using the same `steps.Step` implementations as standalone mode
  - Git tokens and health credentials stay in the agent cluster (no forwarding to control plane)
- Control plane changes:
  - `kardinal-controller` in distributed mode (`--mode=control-plane`) only reconciles Pipelines, Bundles, PolicyGates, and Graphs
  - Step dispatch: sets `kardinal.io/shard` label on `PromotionStep` objects based on Pipeline `spec.shard` field
  - Agent liveness: control plane tracks agent heartbeats; marks shard as unavailable if no heartbeat for 60 seconds
- `config/agent/` Helm chart for the agent binary
- `--mode=standalone` (default) preserves Phase 1 behavior (single binary handles everything)
- Agent reconnect: exponential backoff reconnect to control plane on disconnect
- Documentation: `docs/distributed.md` describing shard topology and credential isolation

## Stage 16: Custom Promotion Steps via Webhook

**Goal:** Implement the custom step dispatch mechanism so teams can inject arbitrary logic into the promotion sequence via HTTP webhooks.

### Deliverables

- Custom step dispatch in `PromotionStepReconciler`:
  - Any `spec.steps[].uses` value not matching a built-in step is treated as a custom step
  - Dispatches HTTP POST to `spec.steps[].webhook.url` with JSON body: `{bundle, environment, inputs, outputs_so_far}`
  - Expects JSON response: `{result: "pass|fail", outputs: {key: value}, message: string}`
  - Timeout: `spec.steps[].webhook.timeoutSeconds` (default 300)
  - Retry: 3 attempts with 30-second backoff on transient HTTP errors (5xx)
- Webhook authentication: `spec.steps[].webhook.secretRef` — Kubernetes Secret with `Authorization` header value
- Step output accumulator: outputs from each step are merged and available to subsequent steps via `${steps.<name>.outputs.<key>}` CEL expressions in later step inputs
- `examples/custom-step/` with a sample custom step HTTP server (Go) and Pipeline referencing it
- Documentation: `docs/custom-steps.md`

### Dependencies

- Stage 6

### Acceptance Criteria

- E2E test: custom step server returns `{result: "pass", outputs: {test_run_id: "abc123"}}` ; next step receives `test_run_id` in its inputs
- Custom step returning `{result: "fail"}` sets PromotionStep to `Failed` and Bundle to `Failed` with the step message as reason
- Webhook with expired secret causes step to fail with `Unauthorized` reason (not retried)
- Timeout: step exceeding `timeoutSeconds` is marked `Failed` with `DeadlineExceeded` reason
- `make test` passes
